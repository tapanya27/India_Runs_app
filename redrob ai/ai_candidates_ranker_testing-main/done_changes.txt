================================================================================
EXTRA — improvements made to meet the project's actual GOAL (resources/)
================================================================================

* Anti keyword-stuffer trap (the core thing the challenge tests):
  _role_fit_multiplier in challenge_ranker.py. The JD states that a candidate
  with all the AI keywords but an unrelated title ("Marketing Manager") is NOT a
  fit. Role is classified from current_title + career-history titles ONLY (the
  headline and skills list are gameable and are deliberately stuffed in the
  dataset). Real career-history evidence can lift an off-title candidate. Effect
  on the 3,000-candidate test: unrelated-title candidates in the top 20 dropped
  from 8 to 0; the top ranks became AI Specialist / ML Engineer / Data Scientist
  / AI Research Engineer / Senior SWE (ML), all in the JD's 5-9 year band.

* Model choice (performance / compute constraint): switched the embedder to the
  compact local model all-MiniLM-L6-v2 (384-dim) and the reranker to
  ms-marco-MiniLM-L-6-v2. Rationale: all-mpnet-base-v2 measured ~1.4 texts/sec
  on this CPU (~20 hours for the 100k build) which is infeasible and violates the
  spec's CPU-only / 5-minute intent. MiniLM is exactly the "compact local model"
  the submission spec recommends. config.py EMBEDDING_DIM updated to 384;
  build_index.py imports the model/dim from config so they never drift.

* submission_metadata.yaml created at repo root from the provided template
  (submission_spec.md Section 10 requires it).


================================================================================
POST-REVIEW FIXES (from an adversarial multi-agent code review of the changes)
================================================================================

A multi-agent review (4 lenses: correctness / edge-cases / performance /
consistency, each finding adversarially verified) was run on the modified code.
Confirmed real defects were fixed:

[FIXED, HIGH] Boundary tie-break admitted the wrong candidate at the top-100
   cutoff. The pool was sorted with reverse=True (candidate_id DESCENDING within
   tied scores) and sliced BEFORE re-sorting, so a tie straddling rank 100 kept
   the largest candidate_id instead of the smallest (spec requires ascending).
   Fix: single sort key=(-final_score, candidate_id) applied BEFORE slicing, so
   both membership and ordering match the spec. (challenge_ranker.py Step 5/6)

[FIXED, HIGH/PERF] skill_matcher Layer-4 ran an unbatched cross-encoder predict()
   per retrieved candidate (~hundreds of model calls inside the rank loop).
   Fix: SemanticSkillMatcher gained use_cross_encoder_validation; the challenge
   ranker constructs it with that OFF (the SBERT probable-match threshold already
   gates these). Adds margin to the 5-minute budget. (skill_matcher.py,
   challenge_ranker.py). The end-to-end ranking step is measured ~140s (< 300s).

[FIXED, LOW] rank.py now fails fast (exit 1) if fewer than top_k results are
   produced, instead of writing an invalid (<100-row) CSV.

[FIXED, LOW] _score_skills_jd_aware used s["name"] subscript on optional keys;
   now uses s.get("name") with a guard.

[FIXED, LOW] Removed the ambiguous "cv" -> "computer vision" skill alias: the
   JD's disqualifier line ("...only CV, speech, or robotics...") was injecting
   "computer vision" into the JD skill set. (config.py)
