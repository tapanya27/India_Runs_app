1) challenge_ranker.py   — the core retrieve -> rerank -> score -> top100 pipeline
================================================================================

1.1  Module docstring updated: pipeline now lists "cross-encoder rerank"; weights
     block updated to the optimized values; noted CE rerank + role-fit.

1.2  Scoring weights changed (item 17, "recommended starting point"):
        W_SEMANTIC    0.30 -> 0.35
        W_SKILL       0.25 -> 0.30
        W_EXPERIENCE  0.20 -> 0.15
        W_PROJECT     0.15 -> 0.10
        W_EDUCATION   0.05 -> 0.03
        W_REDROB      0.05 -> 0.07

1.3  New constants for cross-encoder reranking (item 5):
        USE_CROSS_ENCODER = True
        CE_BLEND = 0.55      # blend weight: CE prob vs FAISS cosine
     FAISS_RETRIEVE kept at 700 (item 1: large recall pool); comment clarified.

1.4  New constants for the role-fit / anti-stuffer defense:
        STRONG_ROLE_TERMS, GOOD_ROLE_TERMS, WEAK_ROLE_TERMS, RELEVANT_WORK_TERMS

1.5  New function _candidate_ce_text(candidate): compact (headline + title +
     summary + top skills + most-recent role) text for the cross-encoder.

1.6  New function _cross_encoder_rerank(jd, candidates, faiss_scores):
     runs the cross-encoder over the retrieved pool and returns, per candidate,
     blended = CE_BLEND*sigmoid(CE) + (1-CE_BLEND)*normalized_faiss_cosine.

1.7  New function _role_fit_multiplier(candidate)  [anti keyword-stuffer trap]:
     - Classifies the role from current_title + career-history TITLES only
       (NOT the headline/skills, which the dataset's stuffers deliberately game).
     - Boosts fit when career-history DESCRIPTIONS show real JD-relevant work.
     - Extra-demotes "unrelated title + many AI skills + no career evidence".
     - Returns a multiplier in [~0.30, 1.0].

1.8  rank_candidates_challenge():
     - Signature gained:  use_cross_encoder: bool = True
     - Step 1b inserted: min-max normalize FAISS cosines, optionally CE-rerank,
       then re-normalize the (blended) semantic score to [0.30, 1.0]. (Replaced
       the old single normalize-only block.)
     - SemanticSkillMatcher now built with use_cross_encoder_validation=False
       (avoids hundreds of per-candidate CE calls in the rank loop).
     - role_mult = _role_fit_multiplier(candidate) added to the modifier chain:
         final = raw * honeypot_mult * consult_mult * role_mult * india_mult
     - role_mult added to the per-candidate all_scores dict.

1.9  _score_skills_jd_aware(): defensive read
        [s["name"] for s in skill_info]  ->  [s.get("name") for s in skill_info if s.get("name")]
        (+ early return if none).

1.10 _build_reasoning():
     - Added `company` from profile.current_company.
     - Added a "relevant work at <company>: ..." clause from JD-relevant terms
       actually present in career descriptions (item 15); else "currently at ...".
     - Removed the duplicated local RELEVANT_WORK_TERMS (now module-level).
     - Added an honest gap: "role/title not aligned with an AI-engineering JD"
       when role_mult is low and there is no relevant career evidence.

1.11 TIE-BREAK BUG FIX (found by review, HIGH):
     Old Step 5 sorted with key=(final_score, candidate_id), reverse=True (ties
     by candidate_id DESCENDING) and sliced top_k BEFORE re-sorting, so a tie
     straddling rank 100 admitted the largest candidate_id instead of the
     smallest (spec requires ascending).
     New: a single canonical sort BEFORE slicing -
        scored.sort(key=lambda x: (-x["final_score"], x["candidate_id"]))
        top = scored[:top_k]
     Membership AND ordering now match the spec at the boundary. Removed the
     redundant second sort.

================================================================================
3) config.py   — models, dimensions, skill aliases/taxonomy
================================================================================

3.1  SBERT_MODEL:        "all-mpnet-base-v2"  ->  "all-MiniLM-L6-v2"
     Reason: all-mpnet-base-v2 measured ~1.4 texts/sec on this CPU (~20 h for the
     100k build) — infeasible and against the spec's CPU-only / 5-minute intent.
     MiniLM is the "compact local model" the spec recommends (~15x faster).

3.2  CROSS_ENCODER_MODEL: "cross-encoder/ms-marco-MiniLM-L-12-v2"
                       -> "cross-encoder/ms-marco-MiniLM-L-6-v2"   (faster).

3.3  EMBEDDING_DIM:      768 -> 384   (all-MiniLM-L6-v2 output dimension).

3.4  Removed the SKILL_ALIASES entry  "cv": "computer vision"  (review, LOW):
     "cv" is ambiguous ("curriculum vitae") and the JD's disqualifier line
     ("...only CV, speech, or robotics...") was injecting "computer vision" into
     the JD skill set. Added a comment explaining the deliberate omission.

8b) FOLLOW-UP FIX — unified job description across CLI and UI
================================================================================

Symptom: the Streamlit output and the CLI submission.csv disagreed (~33% overlap)
because app.py and rank.py each defined their OWN, different DEFAULT_JD (838 vs
1418 chars) -> different JD embedding -> different ranking.

Fix (single source of truth):
  - challenge_ranker.py: added the canonical DEFAULT_JD constant.
  - rank.py: now imports DEFAULT_JD from challenge_ranker (removed its own copy).
  - app.py:  now imports DEFAULT_JD from challenge_ranker (removed its own copy).
Result: the UI and the CLI rank against the identical JD and produce identical
top-100 output. (No index rebuild needed — the JD only affects query time.)
